# 決定パッケージ ⑥: 会議→Handoff 自動ルーティングの live 化（ENABLE_MEETING_ROUTING）

## 1. 決める問い（1つ）
3社合議の Synthesis から Handoff を**自動起票する**ルーティングを、dry-run から **live（実起票）へ切り替える**か。

## 2. 結論ファースト（推奨）
**A. 条件付き Go（カナリア）。** コードは live 前提条件3点（二重起票防止・env/例外テスト・順序バイアス是正、PR #25）を満たし、例外は合議ループを壊さない設計。ただし**本番 Notion への自動書込み挙動が変わる**ため、まず低頻度・監視付きのカナリアで開始し、二重起票・誤ルートが無いことを確認してから常用。

## 3. 現状の事実（検証済み）
- `consensus.py`: `dry_run = not _env_truthy(os.environ.get("ENABLE_MEETING_ROUTING"))`。**env 未設定＝dry-run**（現状）。
- ルーティング呼び出しは try/except で全面ガードされ、**失敗しても合議ループは止まらない**（ログのみ）。
- live 前条件3点は PR #25 で導入済み（引き継ぎ書）。`doc_task` は合議ループに入れない設計。
- 重複起票の歴史的リスク: 窓口OS の未照会起票で CI green タスクが3件重複（Notion 制約リスト 2026-06-15）。→ 起票時の重複チェックが要。

## 4. 選択肢（メリット / デメリット）
| 選択肢 | メリット | デメリット |
|---|---|---|
| **A.（推奨）条件付き Go（カナリア）** | 会議→実装の接続が自動化／例外は合議を壊さない／前条件は実装済み | 本番 Notion へ自動起票が始まる＝要監視。Render env 設定が必要 |
| B. フル live 即時 | 即・全自動 | 二重起票/誤ルートが一気に本番反映されるリスク |
| C. dry-run 維持 | 現状維持・リスクゼロ | 会議結果の Handoff 化が手動のまま |

## 5. リスクと前提・依存
- リスク: **二重起票**（重複チェック必須）・**誤ルート**（型判定ミス）・想定外の Handoff 量産。
- 前提・依存: Render env `ENABLE_MEETING_ROUTING` の設定（インフラ）。デプロイ元ブランチが本番系であることの確認（⑤と整合）。

## 6. 必要な人間 / インフラ作業（エージェント代行不可）
- **Render env `ENABLE_MEETING_ROUTING=1` の設定**（カナリア中は監視）。
- カナリア期間の結果レビュー（起票が妥当か／重複が無いか）。
- エージェント代行可: 重複チェック強化やルート判定テストの追加 PR、カナリア観測手順の docs 化。

## 7. 影響範囲
- Render 設定変更: **あり（env）** / Notion スキーマ変更: なし / 本番挙動変更: **あり（Handoff 自動起票）** / 新規 Cloud Routine: なし

## 8. ロールバック手順
- `ENABLE_MEETING_ROUTING` を未設定/false に戻すと即 dry-run へ。誤起票分は Handoff を Archived/削除で回収（スキーマ不変）。

## 9. Go したら最初の一手
- カナリアとして、まず**重複チェック強化＋ルート判定のテストを足す PR**（エージェント・可逆）を用意 → その上で矢嶋さんが Render env を1日だけ live にし、起票結果をレビュー → 妥当なら常用、問題あれば即 false に戻す。
