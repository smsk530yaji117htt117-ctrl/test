## L-001 マージは「未マージ」を既定とし、git fetch + PR API で事実確認してから記録する
- **詰まり**: 「UI でマージ済み」と伝えられても、`git fetch` では本番ブランチ head が変わっておらず、PR は `state=open / merged=false` のままだった（2026-06-21、#30〜#35 で発生）。
- **なぜ効く**: マージ済みと誤って記録すると、本プロジェクトが明示的に戒める「AI が merge 済みと虚偽報告する」失敗（Notion 制約リスト #24 / LL-119）に直結する。
- **次回どうする**:
  1. マージ報告を受けても「未マージ」を既定として扱う。
  2. `git fetch origin` →本番ブランチ head の sha 変化を確認、かつ GitHub PR API で `merged=true` / `merge_commit_sha` を確認。
  3. 両方で確認できるまで、セッションログ等に「マージ→反映」と書かない。「PR作成・CI green・マージ承認待ち」までを事実として書く。

## L-002 自走の燃料＝監査在庫を、着手前に確認する
- **詰まり**: 「コードだけで潰せる負債」は一気に消化され、数PRで尽きる（本セッションで監査 High/Med/Low の code-only 項目をほぼ完了）。在庫を見ずに走ると途中で燃料切れになる。
- **なぜ効く**: 改善エンジンは「発見（燃料）」が律速。実装能力より先に在庫が切れる。
- **次回どうする**: 着手前に `research/code_audit.md`（baseline）と `tools/audit_scan.py` の出力（差分監査）で**未消化項目の在庫数**を数える。在庫が薄ければ、実装より先に「しくみ1（継続監査）」で燃料を補充する。

## L-003 コード完結項目はすぐ尽きる→人間判断項目の「前裁き」が次の律速
- **詰まり**: code-only 項目が尽きた後に残るのは ④体重ゼロ化 / ⑤ブランチ分岐 / ⑥meeting routing live化 など、戦略・インフラ・スキーマを伴う**人間判断待ち**。ここで停滞する。
- **なぜ効く**: 人間の判断コスト（材料集め・選択肢比較）が摩擦になり、承認ゲートで詰まる。
- **次回どうする**: 人間判断項目は「決定パッケージ」（材料・選択肢・推奨・リスク・前提・ロールバックを1枚に）まで**前裁き**して届ける（しくみ3）。判断を"選ぶだけ"にする。テンプレ: `docs/decision-package-template.md`。

## 運用ノート（手順の再現性メモ）
これらは「失敗」ではなく、再現に効いた段取り。

- **独立PRは本番基点から個別ブランチで切る**（スタックしない）。同一ファイルを触る PR は順次マージ時に軽微な rebase が出うるので、ファイルの重なりを事前に把握する。
- **ローカルCI相当**: `venv` に `pytest pytest-asyncio` を導入すれば、CI（`.github/workflows/ci.yml`）と同じ `python -m pytest` をローカルで再現できる。push 前に green を確認する。
- **PR作成/CI確認**: `gh` 不在でも、Git Credential Manager の保存資格情報を `git credential fill` から取り出し（**値は表示しない**）、GitHub REST API で PR 作成・check-runs 確認ができる。
- **非同期テスト**: 追加依存を避けるため、コルーチンは `asyncio.run()` で駆動し、重い SDK は `unittest.mock` で差し替える（既存テストと同方針）。
