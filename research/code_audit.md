# PersonalOS — Read-Only Codebase Audit (Final Report)

## Executive Summary

This is a read-only diagnostic audit of the **PersonalOS** personal-automation system on its **production branch `claude/notion-api-setup-BQGwN` (HEAD `64e5055`)** — not `main`/`master`. Every finding below survived an adversarial verification pass against the actual source tree and git history; **1 finding was rejected during verification** and removed, and several others were downgraded after their stated impact or evidence proved overstated (corrections are noted inline). The audit covers the consensus deliberation loop (`consensus.py` / `main.py`, on a Render `*/10` cron), the Notion bridge/relay layer, the read-only dashboard, CI configuration, and the untracked `daytime_ops/` directory. The highest-impact themes are: an operational timeout/guard mismatch that can silently block a queued item for ~52 minutes, missing HTTP timeouts on the hot cron path, zero test coverage on the most critical write paths, and substantial structural divergence across the `master` / `origin/main` / production branches with no single canonical deploy branch. No change was made to any file; this report is diagnosis only.

## Top 5 — Fix First

1. **`notion_utils.py:44` + `relay/create_handoff_page.py:89` — missing `urlopen` timeout** — A network hang on the hot cron path stalls the whole production loop and burns the cron slot; a one-line `timeout=30` matches the existing `bridge_notion.py` convention. *(reversible = safe to automate)*
2. **`consensus.py:47` + `main.py:24` — 480s kill vs 60-min guard mismatch** — A killed run leaves a page silently blocked in `Running` for ~52 minutes with no error recorded; aligning the timeout/guard restores recoverability. *(reversible = safe to automate)*
3. **`consensus.py` `process_one` / `synthesize` fallback — zero test coverage on the critical write path** — The function that writes every production Notion row, and the OpenAI synthesis fallback that fires exactly when Claude is down, are untested; mock-based tests add a safety net without touching runtime behavior. *(reversible = safe to automate)*
4. **`.github/workflows/ci.yml` — branch-scoped, divergent CI gate** — The deploy branch runs a narrower CI than `origin/main`; reconciling it (while preserving `pytest-asyncio`) closes the gate gap, but the change also intersects branch strategy. *(irreversible = needs human approval)*
5. **Branch divergence: `consensus.py`/`render.yaml`/`health/` across `master` vs `origin/main` vs production** — There is no single canonical deploy branch, so the wrong target silently runs the wrong runtime; choosing the canonical branch and merging the module/test sets is a strategic, multi-branch decision. *(irreversible = needs human approval)*
